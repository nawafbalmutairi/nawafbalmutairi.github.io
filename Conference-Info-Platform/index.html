Context
As a software architect at Northumbrisoft, the task was to build a REST API backend that serves conference data to multiple frontend applications: web, mobile, AR headsets, and digital displays. The API provides structured access to the CHI 2023 conference dataset — including research papers, authors, affiliations, content types, and awards. Part 2 extended this into a full microservices architecture using the Kruchten 4+1 model for a multi-conference platform.

— Part 2: Kruchten 4+1 Architecture (Multi-Conference Platform)
Part 2 extended the Conference-Info concept into a multi-conference management platform using microservices architecture. The design was documented using the Kruchten 4+1 View Model — Logical, Process, Development, Physical, and Scenarios — with diagrams created in Eraser.io.
View 1
Logical View
Domain-driven design defining core entities and their relationships.
Conference: Aggregate root managing title, description, date, location
Session: Dependent entity with speaker, start/end times
User: Authentication, authorisation, RBAC profile management
Note: User- and session-agnostic content management
AnalyticsEvent: Persistent event logging for auditing
View 2
Process View
Runtime interactions and communication between microservices.
API Gateway: Token validation, traffic routing, rate limiting
Auth Service: OAuth2, OpenID Connect, JWT tokens
Conference / User / Note Services: RESTful CRUD APIs
Analytics Pipeline: Kafka event streaming, Spark aggregation
Communication: gRPC (sync) + Kafka (async) + Istio service mesh with mTLS
View 3
Development View
Modularity, CI/CD, testing, and code quality standards.
Repos: Separate repositories per microservice + shared-libs
CI/CD: GitHub Actions — lint, test, SonarQube, Pact contract testing
CD: Canary deployments with manual production approvals
Quality: 80%+ unit test coverage, ESLint, Checkstyle
Integration: Testcontainers for realistic environment emulation
View 4
Physical View
Cloud-native deployment infrastructure on AWS.
Compute: AWS EKS with horizontal pod auto-scaling
Database: AWS RDS PostgreSQL — isolated DB per conference + read replicas
Load Balancer: AWS ALB for traffic distribution
CDN: CloudFront for global static asset delivery
Observability: Prometheus + Grafana, ELK stack, Jaeger tracing
+1: Scenarios (Use-Case View)
The scenarios tie all four views together by demonstrating how they support real user workflows
+1 View
The use-case scenarios validate the architecture by tracing user journeys across all four views — from a conference attendee logging in (Auth Service in Process View), browsing sessions (Conference Service querying the Logical View entities), taking notes (Note Service persisting to the isolated Physical View database), through to the entire request flowing through the API Gateway, Kubernetes pods, and monitoring stack. Each scenario proves that the architectural decisions in one view are supported by the infrastructure and design in the others.
